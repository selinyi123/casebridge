## 1. 项目背景

### 1.1 现实断层

- **课堂学的是**：个案工作、小组工作、社区工作、人在情境中、生态系统理论、优势视角、案主自决、专业伦理。
- **真实工作里遇到的是**：走访记录散、个案跟进断、资源链接靠经验、服务计划模板化、督导复盘不足、成效难证明。

### 1.2 行业现实

- 截至 2022 年底，全国已建成乡镇/街道社工站 2.9 万余个，6 万余名社会工作者驻站服务，覆盖率约 77.7%。
- 已形成"政社合作""三社/五社联动"等本土服务模式。
- 基层社工的核心工作是**围绕服务对象的连续专业服务**，而非单纯办公或台账。

### 1.3 项目初衷

> 把社会工作专业知识转化成一个真实可用的数字化服务系统。

不是炫技，不是套壳 ChatGPT，不是办公提效工具。

## 2. 产品定位

### 2.1 一句话定位

**案桥 CaseBridge：面向中国社会工作场景的 AI-Native 个案服务与资源协同系统。**

围绕服务对象的连续支持过程，整合个案管理、需求评估、资源协同、服务计划、转介跟踪、督导复盘和成效追踪，让社工从零散记录和经验判断，升级为可持续、可协同、可复盘的专业服务流程。

### 2.2 它是什么 / 不是什么

| 它是 | 它不是 |
| --- | --- |
| AI-Native 个案服务系统 | 社工版 ChatGPT（只问答，不进业务闭环） |
| 中国本土化 Casebook | 社工办公助手 / 写材料工具 |
| 社工服务连续支持平台 | 政务台账 / 报表 / 验收系统 |
| 资源协同与成效追踪系统 | AI 自动社工 / 自动决策系统 |
| 人机协同专业系统 | 国外 Casebook 的简单复制品 |

### 2.3 与现有系统的差异

现有系统多偏**政务管理、公益 CRM、社工站台账、项目管理或传统个案记录**。CaseBridge 的差异在于：把中国社工业务模型、个案服务闭环、资源协同、督导复盘、成效追踪和 **AI 专业增强内核**整合到一个 AI-Native 系统中。

### 2.4 参考对象（参考，不照搬）

- **Casebook**（human services software）：intake → case management → assessment → reporting 的可配置人类服务平台。
- **CiviCRM CiviCase**：跟踪组织与服务对象之间一组连续互动、流程、任务、后续活动——印证"个案不是一条记录，而是一组连续互动"。

本土化核心：中国社工站 + 个案/小组/社区三大方法 + 三社/五社联动 + 本地资源图谱 + AI 专业增强内核 + 隐私合规。

## 3. 目标用户

| 角色 | 画像 | 第一版是否覆盖 |
| --- | --- | --- |
| 新手社工 / 驻站社工 | 需要评估辅助、资源线索、服务计划草稿、督导式反思 | ✅ MVP 主用户 |
| 社会工作专业学生 | 通过虚拟案例练习评估与计划、接受 AI 反馈 | ⚪ V0.3 |
| 督导 | 复盘服务过程、把关专业判断与伦理 | ⚪ V0.4 |
| 社工站 / 机构管理员 | 资源库维护、权限、审计 | ⚪ V0.4 / V1.0 |

**MVP 默认角色 = 单个社工（主责社工）**，权限体系先做最小集（见 §10）。

## 4. 核心目标（6 项）

| 目标 | 说明 |
| --- | --- |
| 1. 让个案服务连续 | 服务对象从接案到结案有完整时间线 |
| 2. 让需求评估专业 | AI 按社工框架提示遗漏，不替社工下结论 |
| 3. 让资源链接可执行 | 把民政、残联、妇联、司法、医疗、志愿者、公益等资源结构化 |
| 4. 让服务计划可追踪 | 每个目标都有行动、责任、时间、状态 |
| 5. 让督导复盘有依据 | 督导看过程与专业判断，不只看结果 |
| 6. 让成效评估有证据 | 服务变化、转介结果、跟进记录可回溯 |

## 5. 服务主线（必须作为项目主轴）

```
服务对象进入
    ↓
建档 / 关系建立
    ↓
接案记录
    ↓
AI 结构化理解
    ↓
需求评估
    ↓
资源匹配
    ↓
服务计划
    ↓
介入 / 跟进记录
    ↓
转介跟踪
    ↓
风险变化追踪
    ↓
督导复盘
    ↓
成效评估
    ↓
结案 / 长期支持
```

**第一性原理：社工如何更好地服务人。** 办公、台账、报表、验收材料可作为第二层能力，不能喧宾夺主。

## 6. MVP 范围：独居老人个案服务闭环

### 6.1 为什么只做一个闭环

> 如果只做"大平台架构图"，就是空谈。第一版必须做一个可演示、可跑通的完整闭环。

### 6.2 场景设定

- **站点**：青桥街道社工站（和安社区）
- **服务对象**：C-0001，78 岁独居老人
- **主要问题**：餐食困难、行动不便、家庭支持不足、睡眠不好

### 6.3 MVP 必须跑通的 9 步

1. 新建服务对象档案
2. 输入入户走访记录
3. AI 提取需求与风险线索
4. AI 按社工框架提示遗漏
5. 系统匹配社区资源
6. 生成服务计划草稿
7. 记录转介状态
8. 生成督导式复盘问题
9. 追踪服务目标变化

### 6.4 验收演示流（项目第一版成立的判据）

```
打开系统
    ↓
进入 C-0001 独居老人档案
    ↓
查看入户走访记录
    ↓
点击 AI 分析
    ↓
看到需求、风险线索、遗漏信息
    ↓
点击匹配资源
    ↓
看到长者饭堂、卫生服务中心、志愿者探访队
    ↓
生成服务计划草稿
    ↓
人工确认
    ↓
生成跟进待办
    ↓
记录一次转介结果
    ↓
AI 生成督导复盘问题
    ↓
成效页显示目标变化
```

只要这个闭环顺畅，MVP 就成立。

## 7. 非目标范围（第一版禁止项）

第一版**不做**以下内容（违反即偏离 MVP）：

- 真实个案导入
- 微信小程序
- 全国资源库
- 多机构 SaaS / 复杂多租户
- 复杂权限矩阵
- 自动风险评级（AI 不下结论）
- 自动转介（AI 不替社工链接）
- 自动救助资格判断
- AI 直接联系服务对象
- 大型报表中心
- 复杂知识图谱 / 向量检索（先用标签规则）

详见 §18。

## 8. 功能模块（产品骨架）

```
CaseBridge
├── 01 服务对象中心        个人/家庭/关系网络/社区/风险线索/服务历史
├── 02 个案工作中心        接案→预估→计划→介入→跟进→评估→结案→回访
├── 03 资源协同中心        社区/政策/公益/志愿/医法教就资源 + 转介跟踪
├── 04 服务计划中心        目标/行动/责任人/时间/资源/风险预案/评估指标
├── 05 转介与跟进中心      待核实→已联系→已转介→成功/失败→待回访→已完成
├── 06 督导复盘中心        遗漏检查/伦理提醒/督导式问题/框架对照/复盘记录
├── 07 成效追踪中心        目标达成/资源结果/风险变化/参与度/长期跟进
└── 08 AI 专业增强内核     Intake/Assessment/Match/Plan/Follow-up/Supervision/Outcome/Privacy
```

> 模块 08（AI 内核）是**增强层，不是主角**。主角永远是社工服务流程。AI 能做什么/不能做什么见 `docs/ai/ai-rules.md`。

### 8.1 MVP 必做模块

- ✅ 01 服务对象中心（建档 + 时间线）
- ✅ 02 个案工作中心（走访记录 + 个案阶段）
- ✅ 03 资源协同中心（资源库 + 标签匹配 + 转介状态）
- ✅ 04 服务计划中心（结构化目标 + 行动步骤）
- ✅ 05 转介与跟进中心（状态流转）
- ✅ 06 督导复盘中心（督导式问题生成）
- ✅ 07 成效追踪中心（目标状态变化）
- ✅ 08 AI 内核（走访结构化、评估提醒、计划草稿、督导问题、隐私脱敏）

### 8.2 MVP 内置专业框架（必须写进产品结构，不只写在介绍页）

人在情境中、生态系统理论、优势视角、赋权理论、案主自决、保密原则、非评判态度、个别化原则、危机介入边界、任务中心模式。

它们落地为：理论框架库、评估维度库、伦理规则库、风险提示库、督导问题库、服务计划模板库、资源匹配规则库（详见 schema.md 与 ai-rules.md）。

## 9. 页面设计（MVP）

| 页面 | 核心内容 | 目的 |
| --- | --- | --- |
| 工作台首页 | 今日待跟进个案 / 待确认 AI 草稿 / 待复盘 / 资源转介待反馈 / 目标变化提醒 | 一进系统就知道今天该做什么 |
| 服务对象档案页 | 左：基本信息；中：服务时间线；右：AI 评估摘要 | 围绕人的连续档案 |
| 接案/走访记录页 | 文本输入 → AI 结构化输出（需求/风险/优势/遗漏/下一步） | 把零散记录变成结构化理解 |
| 资源匹配页 | 推荐资源 / 匹配理由 / 需核实项 / 状态 | 需求映射到资源 |
| 服务计划页 | 目标/短期/中期/长期 + 行动/责任人/时间/指标 | 评估转化为可执行方案；AI 草稿必须可接受/修改/退回/确认 |
| 跟进时间线页 | 按日期的事件与状态变化 | 防止跟进断裂 |
| 督导复盘页 | AI 督导式问题 + 伦理/缺失提示 | 帮新手形成专业反思 |
| 成效追踪页 | 目标进度 / 资源链接结果 / 风险变化 / 社工反思 / 下一步 | 用证据证明服务成效 |

### 9.1 服务计划页必须的交互

AI 生成草稿后必须提供四个动作：**接受为草稿 / 修改 / 退回 / 提交人工确认**。任何 AI 输出不得绕过人工确认直接成为正式记录（见 privacy-design.md）。

## 10. 权限（RBAC，MVP 最小集）

MVP 只实现最小可演示权限：

| 角色 | MVP 能力 |
| --- | --- |
| `social_worker`（默认） | 建档、写记录、触发 AI、确认草稿、匹配资源、写计划、跟进、复盘 |
| `admin`（演示用） | 同上 + 资源库维护 + 用户管理（演示账号） |

完整 RBAC（学生/督导/机构多租户）延后到 V0.3 / V1.0。MVP 不做组织级数据隔离，但 schema 必须预留 `organization_id` 字段。

## 11. AI 能力清单（详见 ai-rules.md）

| 能力 | 输入 | 输出 | 落库位置 |
| --- | --- | --- | --- |
| AI Intake 接案理解 | 走访/接案文本 | 结构化 JSON（需求/风险/优势/遗漏/下一步） | `ai_outputs` |
| AI Assessment 评估辅助 | 个案信息 | 评估维度遗漏提醒 | `ai_outputs` |
| AI Resource Match 资源匹配 | 需求标签 | 推荐资源 + 匹配理由 | 标签规则 + `ai_outputs` |
| AI Case Plan 计划草稿 | 评估 + 资源 | 结构化服务计划 JSON | `ai_outputs` |
| AI Follow-up 跟进提醒 | 时间线 + 目标 | 待办与提醒 | `ai_outputs` |
| AI Supervision 督导复盘 | 个案记录 | 督导式问题 + 伦理提示 | `ai_outputs` |
| AI Outcome 成效分析 | 目标/资源/跟进 | 变化摘要草稿 | `ai_outputs` |
| AI Privacy Guard 隐私脱敏 | 任意待分析文本 | 脱敏后文本 + PII 命中 | 调用前置，不入正式字段 |

**铁律：所有 AI 输出先进 `ai_outputs`，等待人工确认，绝不直接写入正式业务字段。**

## 14. 隐私与安全底线（详见 privacy-design.md）

### 14.1 MVP 隐私原则

不使用真实姓名 / 身份证 / 手机号 / 真实住址；不接真实未成年人个案；不自动判断风险等级；不让 AI 直接写入正式结论。

### 14.2 数据流（隐私内建）

```
原始输入
    ↓
敏感信息检测（PII）
    ↓
脱敏处理
    ↓
AI 分析
    ↓
AI 草稿输出
    ↓
人工确认
    ↓
正式入库
    ↓
审计留痕
```

### 14.3 必备机制

RBAC 权限、字段级脱敏（身份/电话/地址默认隐藏）、AI 调用日志（输入/输出/模型/用户）、人工确认门禁、操作审计、数据导出控制、**原始记录保留（AI 摘要不覆盖原始记录）**。

## 16. 验收标准

MVP 完成不以功能数量衡量，而以 §6.4 演示流是否顺畅为准。补充功能级验收：

- [ ] 能登录，看到工作台待办
- [ ] 能新建 C-0001 档案并录入走访记录
- [ ] 触发 AI 分析，得到结构化需求/风险/遗漏
- [ ] 资源匹配返回长者饭堂 / 卫生服务中心 / 志愿者探访队
- [ ] 生成服务计划草稿并能人工确认
- [ ] 跟进待办出现在时间线
- [ ] 能记录一次转介结果
- [ ] AI 生成督导式复盘问题
- [ ] 成效页显示目标状态变化
- [ ] 所有 AI 输出均经 `ai_outputs` + 人工确认，未绕过
- [ ] 走访文本中的 PII 被脱敏，AI 调用有审计日志

## 17. 开发阶段与里程碑

| Phase | 周期 | 产出 | 是否接 AI |

## 19. 项目负责人（你）的角色

你不是纯程序员，你是：产品负责人 + 社工专业设计者 + AI 流程设计者 + vibecoding 指挥者 + 项目 loader + 最终决策者。

### 19.1 对 AI 编程工具的硬约束（每次开发都要带上的"系统提示"）

> 你是我的高级全栈工程师。请严格根据 `docs/product/prd.md` 和 `docs/architecture/schema.md` 开发。不得新增未定义功能。不得绕过 RBAC。不得让 AI 输出直接写入正式业务字段。所有 AI 输出必须先进入 `ai_outputs` 表，等待人工确认。遵守 `docs/privacy/privacy-design.md` 的脱敏与审计要求。

### 19.2 任务拆分原则

不要说"帮我做一个社工系统"。每次只给一个小任务：

- "实现 `clients` 表、API 和服务对象列表页面"
- "再实现 `cases`、`case_notes` 表和个案时间线"
- "实现 AI 结构化接口，输入 case_note，输出 JSON 到 `ai_outputs`"

### 19.3 每个功能的验收问题

1. 这个功能是否进入个案服务闭环？
2. 是否保护隐私？
3. 是否保留人工确认？
4. 是否能演示？
5. 是否符合社工逻辑？

不符合就砍。
