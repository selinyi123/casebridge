# CaseBridge 案桥 — 隐私与合规设计

> 状态：v0.1（对应 PRD / schema / ai-rules v0.1）
> 用途：本文是 CaseBridge 的**底线文档**。隐私不是后补功能，而是从第一天内建（Privacy-by-Design）。
> 适用：所有涉及服务对象数据的代码、AI 调用、演示数据、导出，必须符合本文。
> 配套：`schema.md`（clients/ai_audit_logs/audit_logs）、`ai-rules.md`（PII Redactor / Risk Guard）。

---

## 0. 为什么这是底线

CaseBridge 处理的不是普通用户数据，而是高度敏感的社会服务数据，可能涉及：

- 身份证、手机号、家庭住址
- 未成年人信息
- 低保 / 特困 / 残障身份
- 疾病与心理健康情况
- 家庭暴力、照护缺失、司法矫正
- 家庭关系、经济困难

《个人信息保护法》（PIPL）将**特定身份、医疗健康、行踪轨迹、不满十四周岁未成年人信息**等列为**敏感个人信息**，并规定个人信息处理包括收集、存储、使用、加工、传输、提供、公开、删除等**全流程**均受规范。

因此系统的隐私架构必须在 Phase 0 就设计完成，不得以"MVP 先跑起来再说"为由推迟。

---

## 1. Privacy-by-Design 七原则（项目落地版）

| 原则 | 在 CaseBridge 的落地 |
| --- | --- |
| 主动而非被动 | 默认脱敏展示、默认最小字段返回 |
| 隐私作为默认 | 不需用户勾选就保护；明文需主动授权 |
| 隐私嵌入设计 | 脱敏/审计/RBAC 写进 schema 与中间件，非业务代码临时加 |
| 正和而非零和 | 隐私保护不牺牲核心业务闭环可用性 |
| 全生命周期 | 从录入→存储→AI→展示→导出→删除全程覆盖 |
| 可见透明 | 审计日志可查、AI 调用可追溯 |
| 以用户为中心 | 尊重案主自决：同意状态可记录、可撤回 |

---

## 2. MVP 隐私原则（硬约束）

第一版严格遵守：

- ❌ 不使用真实姓名（用匿名编号 `C-0001` 为主标识）
- ❌ 不使用身份证号
- ❌ 不使用手机号
- ❌ 不使用真实住址
- ❌ 不接真实未成年人个案
- ❌ 不做真实自动转介
- ❌ 不做自动风险定性（AI 不下结论）
- ❌ 不让 AI 直接写入正式结论

> 演示与开发全部使用**虚拟/脱敏案例**。

---

## 3. 数据流（隐私内建）

```
原始输入（社工录入走访文本）
    ↓
敏感信息检测（PII Redactor）
    ↓
脱敏处理 → content_clean
    ↓
AI 分析（只读 content_clean）
    ↓
AI 草稿输出 → ai_outputs(payload, review_status=pending)
    ↓
人工确认（社工 review: accept/modify/reject）
    ↓
正式入库（搬运到 case_assessments/case_plans/...）
    ↓
审计留痕（audit_logs + ai_audit_logs）
```

核心原则：**AI 输出只能是草稿，不能直接变成正式个案结论。**

---

## 4. PII 检测与脱敏

### 4.1 识别范围（高敏感字段）

- 身份证号（18/15 位）
- 手机号 / 固话
- 详细住址（到门牌）
- 真实姓名（与匿名编号解耦）
- 未成年信息（年龄 < 14 特别标记）
- 疾病/诊断/残疾证号
- 银行账号

### 4.2 脱敏策略

| 类型 | 策略 |
| --- | --- |
| 身份证 | 正则识别 → 替换 `[REDACTED_ID]`；明文不进模型 |
| 手机号 | 替换 `[REDACTED_PHONE]` |
| 详细住址 | 保留到社区级，门牌替换 `[REDACTED_ADDR]` |
| 真实姓名 | 存 `real_name_encrypted`，展示用 `display_name`/匿名编号 |
| 未成年 | 整条记录标 `minor=true`，MVP 拒绝录入真实未成年个案 |

### 4.3 脱敏前置（ai-rules.md §2）

- 任何文本进模型前必须先脱敏。
- `ai_audit_logs.input_pii_redacted=false` 的调用**直接拒绝并报错**。
- Risk Guard 解析后再次检查残留 PII，命中则拒绝输出。

---

## 5. 访问控制（RBAC）

MVP 最小集（见 prd.md §10）：

| 角色 | 数据访问 | PII 明文 |
| --- | --- | --- |
| `social_worker` | 本机构个案 | 默认脱敏；按需申请 |
| `admin` | 本机构全部 + 资源库 + 用户管理 | 可查看明文（审计） |

- 字段级脱敏：身份/电话/地址在 API 响应中默认隐藏；`?include_pii=true` 需更高权限。
- 预留 `supervisor`/`student`/`org_admin` 角色，V0.3 起启用。
- 多租户：所有表带 `organization_id`，MVP 单租户但查询层强制带该字段，防越权。

---

## 6. 审计与留痕

### 6.1 操作审计 `audit_logs`
谁看过、谁改过、谁导出、谁确认了 AI 草稿，全部留痕：
`view_client` / `export` / `update` / `confirm_ai_output` / `delete` …，含 user / entity / detail / ip / ua / time。

### 6.2 AI 调用审计 `ai_audit_logs`
记录输入是否脱敏、模型、提示词哈希、token、延迟、错误。

### 6.3 原始记录保留
- `case_notes.content_raw` 永久保留，**AI 摘要不得覆盖原文**。
- AI 产出的结构化结果只进 `ai_outputs`，确认后沉淀为新字段，不改写原始记录。

---

## 7. 数据导出控制

- 导出 Word/PDF/Excel 需权限（MVP 仅 admin），并写审计。
- 导出内容默认脱敏；导出明文需二次确认 + 留痕。
- MVP 不实现批量导出真实数据（演示用脱敏种子数据）。

---

## 8. 案主自决与同意

- `clients.consent_status`：`none`/`oral`/`written`/`expired`。
- `case_referrals.client_consent` 默认 `null`，**必须社工核实案主意愿后才能推进转介状态**。
- AI 不得绕过同意状态自动转介。
- 督导复盘把"是否记录案主意愿"列为标准检查项。

---

## 9. 演示与开发数据规范

- 全部使用虚拟案例，禁止真实 PII。
- 种子数据：10 个虚拟 clients（C-0001~C-0010）、30 条虚拟 resources。
- 联系人电话用 `1380000xxxx` 占位，姓名用化名。
- 提示词 few-shot 只用 C-0001 虚拟案例。

## 10. 安全机制清单（落地核对表）

| 机制 | 说明 | 落地位置 |
| --- | --- | --- |
| RBAC 权限 | 学生/社工/督导/管理员权限不同 | schema users.role + 中间件 |
| 租户隔离 | 不同机构数据隔离（预留） | organization_id 全表 |
| 字段级脱敏 | 身份/电话/地址默认隐藏 | Privacy Service + 序列化层 |
| AI 调用日志 | 记录输入/输出/模型/用户 | ai_audit_logs |
| 人工确认 | AI 结果不能直接成为正式记录 | ai_outputs.review_status |
| 操作审计 | 谁看/改/导出都留痕 | audit_logs |
| 数据导出控制 | 导出需权限 | 导出接口 + 审计 |
| 原始记录保留 | AI 摘要不覆盖原文 | case_notes.content_raw |
| 脱敏前置 | 模型只见 content_clean | PII Redactor |
| 同意管理 | 转介需案主同意 | consent_status / client_consent |

---

## 11. 合规边界声明（演示与路演用）

- 本系统是**辅助社工的专业工具，不替代社工专业判断**。
- AI 所有输出为草稿，责任由确认的社工承担。
- 处理真实服务对象数据前，需具备合法依据（机构授权、案主同意、法定职责等），并完成正式合规评估——**MVP 阶段不处理任何真实个案数据**。
- 未成年、家暴、司法矫正等极高敏感场景，MVP 不接入真实数据。
