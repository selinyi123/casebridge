# v0.1.5 Unified Timeline and Audit Checklist

> Scope: make case actions traceable before AI intake is introduced.

## Expected API route

- `GET /api/v1/cases/CASE-0001/timeline`

## Expected event kinds

- `note`
- `goal`
- `resource_link`
- `audit`

## Manual checks

1. Create a case note.
2. Create a service goal.
3. Create a resource link.
4. Update the resource link status with explicit agreement state.
5. Fetch `/api/v1/cases/CASE-0001/timeline`.
6. Confirm the timeline contains note, goal, resource_link, and audit events.

## Audit events expected

- `note.created`
- `goal.created`
- `resource_link.created`
- `resource_link.status_changed`

## Acceptance rules

- No AI provider is called.
- Audit events contain entity type and entity id.
- Timeline merges notes, goals, resource links, and audit events into one ordered list.
- Audit payload must not store full raw sensitive note text.
- This is still MVP audit, not a production compliance log.

## Known limitations

- The web UI still needs a dedicated unified timeline panel if the current workspace display is insufficient.
- Actor is still a demo actor.
- No RBAC enforcement yet.
- Audit events are append-only by convention, not yet protected by database-level immutability.
