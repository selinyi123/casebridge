# v0.1.6 AI Intake Gate Checklist

> Scope: first AI capability under strict draft-only and human-review constraints. Provider is mock only.

## Expected API routes

- `GET /api/v1/cases/CASE-0001/ai/outputs`
- `POST /api/v1/cases/CASE-0001/ai/intake`
- `PATCH /api/v1/cases/CASE-0001/ai/outputs/{output_id}/review`

## Manual API checks

Generate draft:

```bash
curl -X POST http://localhost:8000/api/v1/cases/CASE-0001/ai/intake \
  -H 'Content-Type: application/json' \
  -d '{"note_id":"NOTE-0001"}'
```

Review draft:

```bash
curl -X PATCH http://localhost:8000/api/v1/cases/CASE-0001/ai/outputs/AIOUT-0001/review \
  -H 'Content-Type: application/json' \
  -d '{"review_status":"accepted","reviewer_notes":"manual review completed"}'
```

## Manual web checks

1. Open `/clients/c-0001/ai`.
2. Generate mock AI draft from an existing note.
3. Confirm output appears with `pending` review status.
4. Accept or reject the draft.
5. Confirm status changes.
6. Open `/clients/c-0001/timeline`.
7. Confirm AI draft and review audit events appear.

## Acceptance rules

- No real external model is called.
- Redaction runs before mock provider receives text.
- AI output is stored only in `ai_outputs`.
- AI output starts with `review_status=pending`.
- Review changes only `ai_outputs`, not formal case fields.
- AI draft creation and review both write audit events.
- Formal case fields are not mutated by AI generation or review.

## Known limitations

- Provider is deterministic mock only.
- No prompt registry yet.
- No model-provider abstraction yet.
- No production DLP service yet.
- Accepting an AI draft does not apply it to formal assessment fields; that must remain a later explicit workflow.
