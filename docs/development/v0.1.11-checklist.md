# v0.1.11 Assessment Versioning and Reviewer Role Checklist

## Expected API routes

- `GET /api/v1/cases/CASE-0001/assessments/{assessment_id}/corrections`
- `POST /api/v1/cases/CASE-0001/assessments/{assessment_id}/corrections`
- `GET /api/v1/cases/CASE-0001/outcomes/report`

## Expected web routes

- `/clients/c-0001/assessments/corrections`
- `/clients/c-0001/outcomes/report`

## Acceptance rules

- Social worker can read assessment correction history.
- Admin can create assessment correction audit events.
- Social worker cannot create correction audit events.
- Outcome report requires authentication.
- Outcome report is manual-only and does not use AI scoring.
- Correction history is stored as audit trail, not as destructive overwrite.

## Known limitations

- There is not yet a separate assessment_version table.
- Correction UI is read-only in this version.
- Supervisor user is planned but not seeded yet.
- Outcome report UI currently requires pasting a token.
